Model `var()` reads as values: the `fashionable/var` module, fallbacks, and the `Vars` identity flip.

**New: `fashionable/var`.** A `Var` value is one custom-property read — a name plus an optional fallback — and the bare read (`Var.of('gap')`) doubles as the property's canonical handle. `Var.fallback` (dual) derives site-specific reads from the handle; `Var.name`, `Var.vars`, `Var.isVar`, and `Var.equals` round out the surface. `Var<Name, Type, Fallback>` carries a covariant `Type` slot, `unknown` while undeclared — the landing zone for typed variables, designed in `docs/vars.md` section 7.

**The `Vars` phantom carries identities (breaking).** Every `Vars` parameter — `Calc`, `Color`, `Declaration`, `RuleSet`, `StyleRule`, `MediaRule`, `Stylesheet` — is now a union of `Var` identities instead of string names: `Calc.var('x')` is a `Calc<Var<'x'>>`, and type annotations spelled `Calc<'x' | 'y'>` become `Calc<Var<'x'> | Var<'y'>>`. Names stay the value-level currency: binding records are still keyed by bare name (`Bindings` maps over `Var.Name<Vars>`, preserving key checking), and every `vars()` report still returns a `ReadonlySet` of names. `Stylesheet.mergeAll` now infers its element type whole and extracts the vars (object-typed phantom candidates do not union across array elements the way string literals did); call sites are unaffected.

**Reads lift and fall back.** `Calc.var` and `Color.var` accept a `Var` read as well as the bare-name sugar (`Calc.var('x')` is `Calc.var(Var.of('x'))`), and `Declaration.make` accepts one as a whole declaration value — the honest spelling of `font-family: var(--stack, sans-serif)`, which as literal text dropped the read from the dependency report. A fallback-carrying read renders `var(--name, fallback)`, with the fallback constrained per context: numeric at `Calc.var`, color-valued at `Color.var` (text coerced through `named`, so CSS-wide keywords are rejected), any declaration value — nested reads included — at the declaration level. The read's identity and its fallback chain's flatten into `Vars`.

**Fallback semantics.** A fallback joins the dependency report (`var(--x, var(--y))` reads both names) but never the requirements channel — it is unreachable in every projection that consumes requirements, so `units()`/`idents()` exclude fallback contents. `bind` on the read's own name replaces the whole read and discards the fallback (author substitution wins over cascade defaulting); binding a name read only inside a fallback substitutes there. A fallback does not exempt its read's name from `solve` bindings. Full derivation: `docs/vars.md`.
